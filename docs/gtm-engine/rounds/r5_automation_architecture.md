## Design Principles

1. **If Jim must touch it every time, it is not automated -- it is "AI-assisted."** This document uses honest labels.
2. **Automate the boring parts, not the human parts.** Drafting is automatable. Filming a kid is not. Posting to Reddit authentically is not.
3. **The file system is the database.** At Jim's scale (solo founder, <50 partners, <20 posts/week), files in a git repo beat any SaaS tool. Files are versionable, searchable, and free.
4. **Two cron jobs, not seven.** The R3 critique proved that 5 of the original 7 were either infeasible or faster to do manually. This spec implements only the 2 that work.

### How Jim Reviews and Approves

1. Jim receives GitHub PR notification on his phone (Sunday morning).
2. Opens PR on GitHub mobile or desktop -- reads the batch file.
3. For each post: checks the `[ ] APPROVED` box, or edits the caption directly in the PR.
4. Merges the PR. The approved content is now in the repo.
5. Opens Buffer (browser or app). Copy-pastes approved Instagram posts. Sets schedule times.
6. For TikTok: saves approved captions in phone Notes app for use when filming.
7. Total Jim time: 45-75 minutes (reading + editing + copy-pasting into Buffer).

### Per-Platform Breakdown

#### TikTok
- **What can be automated:** Caption generation, hashtag suggestions, posting time recommendations, content concept ideas.
- **What cannot be automated:** Filming (100% manual), editing in CapCut (manual), uploading to TikTok (manual -- no free API for posting). TikTok does not allow scheduled posting via any free third-party tool. Buffer does not support TikTok on free tier.
- **Workflow:** Jim films 1-2 videos per week with Oliver (30 min). Edits in CapCut (15 min per video). Uploads to TikTok manually with the pre-generated caption (5 min). Downloads the video and re-uploads to Instagram Reels (5 min).
- **Total per video:** ~55 min filming-to-posted. No shortcut exists.

#### Instagram
- **What can be automated:** Caption generation, hashtag sets, scheduling via Buffer.
- **Buffer free tier limitations:** 3 connected channels, 10 posts per channel in queue. Instagram feed posts and carousels CAN be scheduled. **Reels CANNOT be scheduled on Buffer free tier** -- only on the $6/mo Essentials plan. So: feed posts go through Buffer, Reels are posted manually after cross-posting from TikTok.
- **Workflow for feed posts:** Claude generates caption in batch. Jim approves. Jim copy-pastes into Buffer with the image. Buffer posts at scheduled time.
- **Workflow for Reels:** Jim downloads TikTok video (remove watermark via SnapTik or TikTok's own save-without-watermark). Uploads to Instagram Reels manually. Pastes caption from batch file. 5 min per Reel.

#### Reddit
- **What can be automated:** NOTHING. Reddit's API ToS (effective 2023) prohibits automated posting for marketing purposes. Accounts detected using automation tools get shadowbanned. Reddit specifically targets new accounts with promotional content.
- **Manual workflow:** Claude generates draft post text in the weekly batch. Jim reviews for authenticity (critical -- Reddit users detect marketing copy instantly). Jim posts manually from his Reddit account. Jim engages with comments for 48 hours. ONE subreddit per week maximum.
- **Key rule:** The batch file includes Reddit drafts but Jim must rewrite them in his own voice. Claude's Reddit drafts are starting points, not copy-paste content.

#### X/Twitter
- **API costs:** X API free tier allows 1,500 tweets/month (write) but only 1 app and no read access. Basic tier is $100/month. **Not worth it at any scale Jim will reach in Year 1.**
- **Buffer:** Buffer free tier supports X posting and scheduling (10-post queue).
- **Recommendation:** Use Buffer for X scheduling if Jim decides to be active on X. But X is not a priority channel for kids apps -- parent audience skews toward Instagram and TikTok. **Defer X entirely until Month 4+ and only if Jim has bandwidth.**

### Follow-up Tracking

**Manual, not automated.** Here is why: at 5-50 dentists, the CRM sheet takes 3 minutes to scan on Monday morning. The conditional formatting (yellow = due soon, red = overdue) makes it visual. Automating this with Google Sheets API + Claude Code cron requires 2-3 hours of OAuth setup for a task that takes 3 minutes manually. The break-even is never reached at this scale.

**When to upgrade:** At 100+ contacts, add the "Yet Another Mail Merge" Google Sheets add-on (free for 50 emails/day) to send templated follow-ups directly from Sheets. Setup: 1 hour. But that is a Month 6+ problem.

### QR Code Generation

**Tool:** goqr.me (free, no account required) or the `qr` npm package (generates locally).

**URL format for dentist cards:**
```
https://brushquest.app/go?ref=REFCODE
```

Where `REFCODE` matches the dentist's `Ref Code` column in the CRM sheet. Examples:
- `https://brushquest.app/go?ref=drkim`
- `https://brushquest.app/go?ref=smilepd`
- `https://brushquest.app/go?ref=oakpedo`

**How the redirect works (engineering task -- see Section 5):**

The `/go` path on `brushquest.app` is a static HTML page (hosted via GitHub Pages in the `docs/go/` directory) that:
1. Reads `ref` query parameter
2. Constructs a Play Store URL with UTM parameters:
   ```
   https://play.google.com/store/apps/details?id=com.brushquest.brush_quest
     &utm_source=dentist
     &utm_medium=card
     &utm_campaign=REFCODE
   ```
3. Redirects the user to that URL

**QR code generation process:**
1. Jim decides on a ref code for each dentist (add to CRM sheet column H)
2. Go to goqr.me, paste the URL, download QR PNG
3. Send QR PNG to Fiverr designer who places it on the prescription card template
4. Or: use the `qr` CLI tool locally: `npx qr "https://brushquest.app/go?ref=drkim" > drkim-qr.png`

**Tracking attribution:**

Firebase reads UTM parameters from Play Store install referrer data. Jim checks attribution in:
- Firebase Console > Analytics > Acquisition > Campaign report
- Filter by `utm_source=dentist` to see all dentist-driven installs
- Filter by `utm_campaign=REFCODE` to see per-dentist installs

**Caveat (from R3 critique):** UTM attribution on Android is 50-70% accurate. Many installs lose the referrer (direct Play Store searches, shared links, etc.). Accept this limitation. Supplement with a future "how did you hear about us?" in-app prompt if precise attribution matters.

## 5. In-App Engineering Tasks

These are the highest-leverage growth features in the entire GTM plan. They run forever once built. Each one is specified below with enough detail for Jim (or Claude Code) to implement.

### 5A. In-App Review Prompt

**Package:** `in_app_review` (https://pub.dev/packages/in_app_review -- 4.2K likes, maintained)

**Trigger logic:**
```dart
// In streak_service.dart or a new review_prompt_service.dart
// Trigger conditions (ALL must be true):
// 1. User has completed >= 5 brush sessions (totalBrushes >= 5)
// 2. User has NOT been prompted before (SharedPreferences flag)
// 3. At least 3 days since first brush (prevents day-1 spam)
// 4. InAppReview.isAvailable() returns true

// Call from victory_screen.dart AFTER the victory animation completes
// (not during -- don't interrupt the celebration)
```

**Implementation approach:**
1. Add `in_app_review: ^2.0.9` to `pubspec.yaml`
2. Create `lib/services/review_prompt_service.dart`:
   - `shouldPrompt()`: checks the 4 conditions above
   - `promptForReview()`: calls `InAppReview.instance.requestReview()`
   - `markPrompted()`: sets SharedPreferences flag so it only triggers once
3. In `victory_screen.dart`: after the star animation completes, check `shouldPrompt()`. If true, wait 1 second, then call `promptForReview()`.
4. Add analytics event: `logReviewPromptShown()` and `logReviewPromptCompleted()` (note: Android's native review API does not report whether the user actually left a review, so "completed" means the prompt was shown, not that a review was submitted).

**Estimated effort:** 2-3 hours.

**Why this is the #1 growth feature:** Every 5-star review improves Play Store ranking and conversion rate. The in-app prompt catches users at peak satisfaction (just completed a brush, watching victory animation). Google's native prompt is frictionless (no redirect to Play Store). One-time setup, runs forever.

### 5B. Share Card Generation

**Approach:** `RepaintBoundary` + `RenderRepaintBoundary.toImage()` + `Share.shareXFiles()` from the `share_plus` package.

**Package:** `share_plus: ^10.1.4` (https://pub.dev/packages/share_plus)

**Trigger moments (pick 2-3 for v1):**
1. 7-day streak achieved (first milestone parents would share)
2. New hero unlocked (visual wow factor)
3. All monsters in a world defeated (completion milestone)

**Card design:**
The project already has a web-based share card template at `docs/share-card.html`. The in-app version should match this design:
- Dark space background (#0D0B2E)
- Hero image (current equipped hero)
- Achievement text (e.g., "7-Day Streak!" or "Shadow Unlocked!")
- "Space Ranger" title (NOT the child's real name -- privacy)
- Brush Quest logo + small QR code or "brushquest.app" text
- No child photos, no real names, no personal data

**Implementation approach:**
1. Add `share_plus: ^10.1.4` to `pubspec.yaml`
2. Create `lib/widgets/share_card_widget.dart`:
   - A `StatelessWidget` that renders the card visually using existing theme widgets
   - Wrapped in a `RepaintBoundary` with a `GlobalKey`
3. Create `lib/services/share_service.dart`:
   - `generateShareCard(GlobalKey key)`: captures the widget as a PNG using `RenderRepaintBoundary.toImage(pixelRatio: 3.0)` and saves to temp directory
   - `shareCard(File imageFile, String text)`: calls `Share.shareXFiles()` with the image and a short message ("My kid's a Space Ranger! Try Brush Quest -- brushquest.app")
4. In `victory_screen.dart`: show a share button after milestone events. Button triggers card generation and share sheet.
5. Add analytics events: `logShareCardGenerated(milestone: String)` and `logShareCardShared(milestone: String, platform: String)`.

**Estimated effort:** 4-5 hours.

**Why this matters:** This is the only truly viral mechanic in the app. Parent shares card on iMessage/WhatsApp/Instagram stories. Other parent sees it. Downloads app. The share card acts as a micro-advertisement created by a real user. Track the funnel: `share_triggered > share_completed > referral_install`.

### 5C. Referral Tracking (Landing Page UTM Routing)

**Current state:** The `/go` path at `docs/get/index.html` redirects to `brushquest.app/#download`. It does NOT pass UTM parameters.

**Engineering task:** Create `docs/go/index.html` that reads query params and constructs a UTM-tagged Play Store URL.

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Download Brush Quest</title>
  <script>
    (function() {
      var params = new URLSearchParams(window.location.search);
      var ref = params.get('ref') || 'direct';

      // Map ref codes to UTM parameters
      var playStoreUrl = 'https://play.google.com/store/apps/details?id=com.brushquest.brush_quest';
      playStoreUrl += '&utm_source=dentist';
      playStoreUrl += '&utm_medium=card';
      playStoreUrl += '&utm_campaign=' + encodeURIComponent(ref);
      playStoreUrl += '&referrer=' + encodeURIComponent(
        'utm_source=dentist&utm_medium=card&utm_campaign=' + ref
      );

      window.location.replace(playStoreUrl);
    })();
  </script>
</head>
```

## Appendix B: What This Architecture Does NOT Cover

These are explicitly out of scope and should not be attempted in the first 3 months:

1. **Automated social posting to TikTok or Reddit.** No free API exists. Manual posting is the only option.
2. **Real-time analytics dashboard.** The monitoring routine in Section 3 is the dashboard. Building a programmatic one requires paid APIs and ongoing maintenance that is not justified at this scale.
3. **Automated email sequences.** Jim sends <50 outreach emails per month. Gmail + templates + Claude drafts is the right tool. Email automation adds complexity for zero benefit.
4. **CRM beyond Google Sheets.** Until 100+ contacts, a spreadsheet with conditional formatting is superior to any CRM because it has zero setup overhead and Jim already knows how to use it.
5. **Multi-platform analytics aggregation.** TikTok, Instagram, and Play Store each have their own dashboards. Aggregating them into one view requires paid tools (Sensor Tower, data.ai) or custom engineering. The Monday 15-minute check covers this.
6. **A/B testing infrastructure beyond Play Console.** Play Console has built-in A/B testing for store listings. That is sufficient. Do not build custom A/B testing.
